Aakar. Sandboxed introspection — design

Option B · Safe path for trust_remote_code models

Introspect untrusted models without exposing the backend to RCE

Custom-architecture repos (e.g. nvidia/LocateAnything-3B) ship Python that must execute to build the module tree. Rather than run that code in the API process, we run it in a disposable, capability-stripped, offline sandbox — and keep the trusted parts (downloading, parsing, caching) where they belong. Default off; opt-in per deployment.

The one idea that makes this safe

Separate fetching from executing. Downloading a repo's files runs no code, so it can have network. Executing the model code can run anything, so it gets no network at all. Untrusted code never touches secrets, the host filesystem, or the network — only a read-only snapshot and the meta device. The only thing that crosses back is a validated Spec JSON.

01What we're defending against

Model ids are user-supplied, so any visitor can pick which repo's code runs. The threat model is hostile.

02The two-phase flow

Phase 1 is trusted and online but runs no repo code. Phase 2 runs repo code but is fully offline and sandboxed.

● Phase 1 · Trusted · network ON · no code exec
Snapshot fetch — pull only what introspection needs, into an ephemeral dir.
  • huggingface_hub.snapshot_download(repo_id, allow_patterns=["*.json","*.py"]) — config + modeling source, no weights (meta device needs none).
  • Runs in the API process: downloading files executes nothing, so it's safe here.
  • Handles auth/gating/not-found with the existing typed errors. No token is passed downstream.
↓  hand off a read-only directory path
● Phase 2 · Untrusted · network OFF · code executes
Sandboxed build & walk — the only place repo code runs.
  • Disposable container, --network none, HF_HUB_OFFLINE=1, pointed at the snapshot.
  • Worker does AutoModel.from_config(cfg, trust_remote_code=True) under init_empty_weights() → walks the nn.Module tree → builds the Spec.
  • Emits Spec JSON on stdout, or a structured error + non-zero exit. Nothing else can leave.
↓  stdout (bounded, validated)
● Back in the API · Trusted
Validate, cache, return.
  • Re-parse stdout through Pydantic Spec (untrusted output is still untrusted — validate it).
  • Store in the existing DiskSpecCache keyed by (model_id, config_hash) — the costly run happens once.

Why no weights? Aakar already builds on the meta device (init_empty_weights()), so it never needs the multi-GB weight files — Phase 1 downloads only *.json + *.py (kilobytes). This keeps the sandbox tiny and the fetch fast.

04How it fits Aakar's layers

No per-family adapters, no new boundary. We add another Introspector implementation and a fallback decorator — the existing Protocol/Repository patterns absorb it cleanly.

API / Application — unchanged
routes → ArchitectureService → Introspector (Protocol)
The service still asks "give me a Spec for this model_id." It has no idea a sandbox exists.
Infrastructure — composition root picks the implementation
FallbackIntrospector (Decorator / Chain of Responsibility)
Try the fast in-process path; only if it raises UnsupportedArchitecture because of remote code — and AAKAR_ALLOW_REMOTE_CODE is set — fall through to the sandbox.
TransformersIntrospector primary
Existing, in-process, trust_remote_code=False. Handles ~99% of models. Unchanged.
SandboxedIntrospector fallback
Orchestrates: HubSnapshotFetcher (trusted) → SandboxRunner (spawns the container) → parse & validate Spec.
Sandbox boundary — disposable worker image
aakar-introspect-worker (container)
Reuses the same tree-walk + Spec-builder code as the in-process path, imported as a shared module. Runs it with trust_remote_code=True, offline, then prints Spec JSON. This is the only process that ever executes repo code.
Cross-cutting
DiskSpecCache
A Spec is just JSON regardless of how it was produced — cache the sandbox result like any other, so repeats skip the sandbox entirely.

This honours CLAUDE.md's rule that "the introspector is the only place that turns a model id into a Spec" — SandboxedIntrospector is just another implementation of the same Protocol, not a per-architecture branch.

05The worker contract

A dead-simple, one-shot process boundary: id in, Spec (or error) out. Easy to fake in unit tests.

# orchestrator side (trusted) — pseudocode
snapshot = hub.snapshot_download(model_id, allow_patterns=["*.json", "*.py"])
proc = run_sandbox(
    image="aakar-introspect-worker",
    mounts={snapshot: "/snapshot:ro"},
    argv=["--model-id", model_id, "--snapshot", "/snapshot"],
    network="none", read_only=True, cap_drop="ALL",
    user="65534:65534", memory="2g", cpus=1.5,
    pids_limit=128, timeout_s=45,
)
spec = Spec.model_validate_json(proc.stdout)   # validate untrusted output
cache.put(model_id, config_hash, spec)

06Rollout — incremental, low-risk

1
Extract a worker entrypoint
Pull the tree-walk + Spec-builder into a module callable as python -m aakar_api.worker. Pure refactor — zero behaviour change for the in-process path; covered by existing tests.
2
Build the hardened worker image
Multi-stage Dockerfile (CPU torch + transformers + the worker). Non-root user, minimal base. Reuses the pinned-deps discipline already in the repo.
3
Implement SandboxedIntrospector + HubSnapshotFetcher
Snapshot fetch (allowlisted patterns) → spawn the container with the hardening flags → validate stdout. Gated behind AAKAR_ALLOW_REMOTE_CODE, default off.
4
Wire the fallback + cache + errors
Compose FallbackIntrospector in main.py; add IntrospectionFailed/Timeout domain errors + handlers; cache results. Unit tests inject a fake SandboxRunner (no Docker); one integration test (marked, opt-in) runs a real custom-code model.

07Residual risks & honest caveats

  • It may still not render. Custom __init__s sometimes allocate real tensors or call .cuda() instead of honouring the meta device, or import packages not in the image. Those just fail safely inside the sandbox → surfaced as IntrospectionFailed. That's fine — we degrade, we don't crash.
  • Gated / private models won't load. By design the sandbox has no token, so it can't auth. We never hand credentials to untrusted code — gating stays a Phase-1 (trusted) decision.
  • Cold-start latency. First run pays container + torch-import cost (a few seconds). The Spec cache makes every repeat instant; custom-code models are the minority. A warm worker pool is a later optimisation, not a requirement.
  • The sandbox's output is still untrusted. We re-validate stdout through Pydantic and bound its size — a compromised worker can't smuggle a malformed Spec into the app.

Until B ships, revert the flag. config_loader.py:23 currently has trust_remote_code=True in-process — that already executes repo config code in the API process (the unsafe path). It should go back to False; the sandbox becomes the only route that runs remote code.
